Identifying Design Problems with a Visualization Approach of Smell Agglomerations


This website contains complementary data to the Master Dissertation.

Abstract

Design problems are characterized by violations of design principles affecting a software system. Because they often hinder the software maintenance, developers should identify and eliminate design problems whenever possible. Nevertheless, identifying design problems is far from trivial. Due to outdated and scarce design documentation, developers not rarely have to analyze the source code for identifying these problems. Past studies suggest that code smells are useful hints of design problems. However, recent studies show that a single code smell might not suffice to reveal a design problem. That is, around 80% of design problems are realized by multiple code smells, which interrelate in the so-called smell agglomerations. Thus, developers can explore each smell agglomeration to identify a design problem in the source code. However, certain smell agglomerations are formed by several code smells, which makes it hard reasoning about the existence of a design problem. Visualization approaches have been proposed to represent smell agglomerations and guide developers in identifying design problems. However, those approaches provide a very limited support to the identification of specific design problems, especially the ones affecting multiple design elements. This dissertation aims to address this limitation by proposing a novel approach for the visualization of smell agglomerations. We rely on evidence collected from multiple empirical studies to design our approach. We evaluate our approach with developers from both academy and industry. Our results suggest that various developers could use our visualization approach to accurately identify design problems, in particular those affecting multiple program elements. Our results also point out to different ways for improving our visualization approach based on the developers’ perceptions.

Detection Strategy

Coupling measures how much a SPL component provides features to other components, or depends on with features provided by other components. That is, coupling metrics aim at capturing to what extent the SPL components are attached to the whole SPL. Thus, a high coupling suggests that a SPL component is strongly attached to others and, therefore, might indicate violations of the component principles, which are: feature encapsulation [7]; data access restriction [6]; and replaceability [5]. It might also suggest that the component realizes an essential feature to the SPL product configuration, due to the high number of dependencies with other components. On the other hand, a low coupling suggests that the SPL component is weakly coupled to others, which might indicate that those three component principles are properly followed by the component. It might also suggest that the component realizes a feature that is not essential to the SPL product configuration, due to the low number of component inter-dependencies. In order to detect critical components in component-based SPLs, we used a coupling metric-based strategy proposed in the literature [8]. A metric-based strategy combines multiples metrics to detect components that satisfy a property, the component criticality in our case.

Table 1 presents the coupling metrics used by the select strategy. The first column presents the name of each metric. The second column describes the metric.

Table 1: Metrics combined by the selected strategy to detect critical components

Metric Description
Coupling Between Objects (CBO) [1] Number of components coupled to one specific component.
Fan-in (FANIN) [2] Number of dependencies from other components to a specific component.
Fan-out (FANOUT) [2] Number of dependencies from one components to others.
Depth of Inheritance Tree (DIT) [1] Maximum depth of a component in the inheritance tree.
Number Of Children (NOC) [1] Number of children components of a component.
